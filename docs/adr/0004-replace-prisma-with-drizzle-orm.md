# ADR-0004: Replace Prisma ORM with Drizzle ORM via isolated `@nebula-chat/db` library

- **Status:** Accepted
- **Date:** 2026-05-01
- **Deciders:** @itsDaiton

## Context

Before this change `apps/nebula-chat-server` depended on Prisma directly: `@prisma/client`, `@prisma/adapter-pg`, the `prisma` CLI as a devDependency, a `prisma/` directory holding `schema.prisma` and migrations, and five Prisma scripts on the server's `package.json` (`prisma:generate`, `prisma:migrate`, `prisma:deploy`, `prisma:studio`, `prisma:validate`). The generated client was emitted into the server's `node_modules` at every install, the schema was server-local, and the database layer could not be reused by any future package (e.g. a worker, a CLI, or an edge function) without dragging the whole server along with it.

Two architectural pressures pushed against keeping Prisma:

1. **The monorepo is moving toward published, versioned internal libraries** (TICKET-M2, `docs/new-backend/TICKET-M2-db.md`). The DB schema and client need to live in a standalone `libs/db` workspace package (`@nebula-chat/db`) that the server consumes via `workspace:*`, so other apps/services can adopt the same schema without duplication. Prisma's generated-client model and the `prisma generate` build step do not pair cleanly with publishing a thin, dual-format library — the consumer would still need to re-run codegen.
2. **Build cost and edge readiness.** Prisma's codegen is a mandatory step in every install/build (and the engines binary is large). Drizzle is a pure TypeScript SQL builder with no codegen and no runtime engine — the lib is small, tree-shakeable, and friendly to non-Node runtimes the project may target later.

The schema is small (three tables: `users`, `conversations`, `messages` — see `libs/db/src/schema.ts`) and the codebase makes no use of advanced Prisma features (no nested `create`, no Prisma middlewares, no `Prisma.validator`) that would make migration painful. A switch was viable now and only gets harder later.

## Decision

Replace Prisma with Drizzle ORM, encapsulated in a new isolated `libs/db` workspace package (`@nebula-chat/db`). The package exposes a `createDbClient({ connectionString, maxConnections? })` factory that returns `{ db, pool }` — `db` is the typed Drizzle query client (with the schema bound), `pool` is the raw `pg.Pool` retained so the server can call `pool.end()` from its `onClose` hook. The package also exports the Drizzle schema objects (`users`, `conversations`, `messages`) and the `DbClient` / `DbConfig` / `DbTransaction` types. `apps/nebula-chat-server` consumes it via `workspace:*`, instantiates a single client in `src/db.ts`, and decorates the Fastify instance with it through `src/plugins/db.plugin.ts`. Schema drift and migrations are managed by `drizzle-kit`; migration SQL files live in `libs/db/migrations/` and are checked into git. `DATABASE_URL` is the single source of truth, read from `apps/nebula-chat-server/.env` by both the server runtime and `drizzle-kit` (the lib's `drizzle.config.ts` loads that file via `process.loadEnvFile`).

## Alternatives Considered

- **Keep Prisma.** Rejected — the generated-client model fights the goal of a publishable, codegen-free `@nebula-chat/db` library; every consumer would need its own `prisma generate` step. The Prisma engines binary inflates install size, edge-runtime support is limited, and `@prisma/adapter-pg` adds an extra abstraction layer over a connection pool that Fastify already manages cleanly via plugin lifecycle hooks. None of the advanced Prisma features (relation includes, nested writes, middlewares) are used heavily enough to justify the cost.
- **Raw `pg` / `node-postgres` queries.** Rejected — no type-safe SQL builder, no schema definition layer, and migration tooling would need to be hand-rolled (or layered with a separate migration tool like `node-pg-migrate`). The team would re-invent half of what Drizzle gives for free, with worse DX.
- **Knex.js.** Rejected — query builder API has no native TypeScript inference; types must be supplied separately or generated from a parallel source. Drizzle's schema-as-TypeScript design produces inferred types automatically and integrates with `drizzle-kit` for migrations in a single tool.

## Consequences

- **Positive:**
  - DB schema and client are now a standalone, versioned workspace package (`@nebula-chat/db`, dual ESM+CJS via tsup) that any future app/worker/CLI in the monorepo can import via `workspace:*` with zero duplication.
  - No codegen step at install or build time. `pnpm install` is faster; the server's build pipeline shrinks by one `prisma generate` invocation.
  - Migrations are plain reviewable SQL files under `libs/db/migrations/` (e.g. `0000_brief_vin_gonzales.sql`, `0001_make_cached_not_null.sql`), generated by `drizzle-kit generate` and applied with `drizzle-kit migrate`. Code review can see the exact DDL, not a Prisma migration log.
  - Lightweight runtime: only `drizzle-orm` and `pg` ship at runtime; no Prisma engines binary, no generated client folder.
  - `createDbClient` returns both `db` and `pool`, so graceful shutdown is explicit — the server's Fastify plugin (`apps/nebula-chat-server/src/plugins/db.plugin.ts`) registers `closeDb` on `onClose`, ending the pool deterministically.
  - Dual ESM+CJS build (tsup) means the current CommonJS server consumes `dist/index.js` while a future ESM/edge target can consume `dist/index.mjs` from the same package.
- **Negative / Tradeoffs:**
  - Drizzle's relational query API (`db.query.x.findMany({ with: { ... } })`) is less mature than Prisma's `include`/`select`. Complex joins or aggregates require dropping into the SQL builder (`select().from().leftJoin()...`), which is more verbose for nested reads.
  - The migration workflow changes: contributors used to `prisma migrate dev` now run `pnpm --filter @nebula-chat/db db:generate` (produce SQL) and `pnpm --filter @nebula-chat/db db:push` (dev sync) or `db:migrate` (apply files). Documented in `CLAUDE.md`, but it is a learned workflow.
  - The lib publishes to GitHub Packages (`publishConfig.registry: https://npm.pkg.github.com`); consumers outside the workspace would need package-registry auth. Not an issue today (consumed only via `workspace:*`), but worth noting before any external publish.
- **Neutral:**
  - `DATABASE_URL` connection-string format is unchanged (standard PostgreSQL DSN). No environment-variable migration is required for any deploy target.
  - Table shapes (`users`, `conversations`, `messages`) and column names are preserved; existing rows in production are unaffected. The Drizzle schema in `libs/db/src/schema.ts` matches the prior Prisma schema 1:1, with the same FK cascades and indexes (`conv_user_idx`, `msg_conv_idx`).
  - HTTP API contracts are unchanged; OpenAPI spec is unaffected.

## Implementation Notes

- **Files added (in `libs/db/`):**
  - `src/schema.ts` — Drizzle pgTable definitions for `users`, `conversations`, `messages` with FK constraints (`onDelete: 'cascade'`) and indexes (`conv_user_idx`, `msg_conv_idx`).
  - `src/client.ts` — `createDbClient(config)` factory returning `{ db, pool }`; exports `DbClient`, `DbConfig`, `DbTransaction` types. `DbTransaction` is derived from the Drizzle transaction-callback parameter so repositories can accept either a top-level client or a transaction handle.
  - `src/index.ts` — public surface: re-exports `createDbClient`, the type aliases, and all schema objects.
  - `drizzle.config.ts` — points `drizzle-kit` at `./src/schema.ts`, output `./migrations`, dialect `postgresql`, migrations table `__drizzle_migrations` in schema `public`. Loads `apps/nebula-chat-server/.env` via `process.loadEnvFile` (silently tolerates missing file so CI/CD can inject env directly).
  - `tsup.config.ts` — dual ESM+CJS build emitting `dist/index.js` (CJS), `dist/index.mjs` (ESM), and `dist/index.d.ts`.
  - `package.json` — declares `@nebula-chat/db@0.1.0`, runtime deps `drizzle-orm@^0.44.0` and `pg@^8.16.0`, dev deps `drizzle-kit`, `tsup`, `@types/pg`, `@types/node` (catalog), and the `db:generate` / `db:migrate` / `db:push` / `db:studio` scripts.
  - `migrations/0000_brief_vin_gonzales.sql`, `migrations/0001_make_cached_not_null.sql` — generated SQL migration files.
- **Files modified (in `apps/nebula-chat-server/`):**
  - `src/db.ts` — singleton: imports `createDbClient` from `@nebula-chat/db`, instantiates it with `env.DATABASE_URL`, and exports `db: DbClient` and `closeDb`.
  - `src/plugins/db.plugin.ts` — Fastify plugin: `app.decorate('db', db)` and `app.addHook('onClose', closeDb)` for graceful shutdown.
  - All repositories under `src/modules/*/`: `apps/nebula-chat-server/src/modules/conversation/conversation.repository.ts`, `apps/nebula-chat-server/src/modules/message/message.repository.ts` (and any others) — rewritten to use Drizzle's query builder / `db.query.*` API instead of `PrismaClient`.
  - `apps/nebula-chat-server/package.json` — adds `@nebula-chat/db: "workspace:*"`; removes `@prisma/client`, `@prisma/adapter-pg`, the `prisma` devDependency, and the five `prisma:*` scripts.
  - Root `package.json` — removes `@prisma/engines` and `prisma` from `pnpm.onlyBuiltDependencies`.
- **Files removed:** `apps/nebula-chat-server/prisma/` directory in full (`schema.prisma`, generated client, prior migrations).
- **Migrations required:** Schema parity with Prisma's prior schema is preserved, so no data migration is needed. The Drizzle migration history bootstraps with `0000_brief_vin_gonzales.sql` representing the current schema; `0001_make_cached_not_null.sql` tightens `messages.cached` to `NOT NULL` (matching the Drizzle schema). Apply via `pnpm --filter @nebula-chat/db db:migrate`. The `__drizzle_migrations` bookkeeping table will be created on first run.
- **OpenAPI / contract impact:** none. No route shapes, request/response schemas, or HTTP semantics change.
- **Rollback plan:** revert the `feat/m-2-drizzle-db` branch commits. Concretely: restore `apps/nebula-chat-server/prisma/`, re-add `@prisma/client`, `@prisma/adapter-pg`, and `prisma` to the server's `package.json`, restore the `prisma:*` scripts, swap `import ... from '@nebula-chat/db'` back to the Prisma client imports in `src/db.ts` and the module repositories, and remove `libs/db/`. Because `DATABASE_URL` and the underlying tables/columns are unchanged, no DB rollback is required — the schema itself is compatible with both ORMs.
